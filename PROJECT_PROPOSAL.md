## 🎯 项目主题

### OrbitalGuard - 近地轨道空间碎片与卫星交通管理系统

该数据库应用程序旨在管理和分析日益拥挤的地球轨道环境。随着商业航天（如 Starlink、OneWeb）的爆发式增长，轨道碰撞风险急剧增加，空间碎片问题已成为全球航天安全的核心挑战。

#### 项目背景

目前，近地轨道上约有：
- **8,000+** 活跃卫星在轨运行
- **34,000+** 可追踪的空间碎片（>10cm）
- **1,000,000+** 微小碎片（1-10cm，无法完全追踪）

2023年国际空间站（ISS）执行了**6次紧急规避机动**以躲避碎片，商业卫星运营商面临日益严峻的空间态势感知（SSA）挑战。

#### 项目目标

该系统将整合以下核心数据：
- **卫星实时位置数据**（TLE - Two-Line Elements）
- **物理属性**（质量、预期寿命、雷达截面积）
- **发射历史与寿命周期**（发射日期、运营商、当前状态）
- **碎片群轨道分布**（通过物体名称识别碎片群，分析其轨道参数）

通过该系统，我们将协助：
- **卫星运营商** 进行防撞预警和轨道维持决策
- **发射服务商** 规划"净空"发射窗口
- **监管机构** 评估轨道环境的可持续性
- **研究人员** 分析长期碎片演化趋势

#### 技术特点

本项目将涉及：
- **时空数据处理**：三维轨道坐标与时间序列的耦合查询
- **复杂几何计算**：碰撞检测算法（最小距离、相对速度）
- **聚合与窗口函数**：轨道密度统计、碎片群统计分析
- **子查询与自连接**：物体对比分析、碰撞风险评估
- **动态参数化查询**：基于用户输入坐标的可见性计算

## 📊 数据源

### 1. CelesTrak (NORAD Data)
提供所有公开卫星和碎片的轨道根数（TLE）和目录数据。
- **URL**: [https://celestrak.org/NORAD/elements/](https://celestrak.org/NORAD/elements/)
- **数据格式**: TLE（Two-Line Element Set）
- **更新频率**: 每日更新
- **内容**: 轨道参数（半长轴、偏心率、倾角、升交点赤经等）

### 2. Space-Track.org (USSTRATCOM)
提供更详细的卫星目录、发射信息和衰减预测。
- **URL**: [https://www.space-track.org/](https://www.space-track.org/)
- **数据格式**: JSON/CSV（需注册API）
- **更新频率**: 实时更新
- **内容**: NORAD ID、国际编号、发射场、衰减预测、轨道寿命

### 3. Union of Concerned Scientists (UCS) Satellite Database
提供卫星的详细功能、用户、承包商和预期寿命数据。
- **URL**: [https://www.ucsusa.org/resources/satellite-database](https://www.ucsusa.org/resources/satellite-database)
- **数据格式**: Excel/CSV
- **更新频率**: 季度更新
- **内容**: 卫星名称、用途、运营商、质量、预期寿命、轨道类型

### 说明：数据分析方法

本项目通过分析 **TLE快照数据** 和 **卫星目录** 来研究碎片分布：
- **碎片识别**: 通过物体名称中的 "DEB" 标识和NORAD ID前缀识别碎片群
- **碎片归属**: 通过国际编号（International Designator）中的发射任务前缀关联碎片与原始卫星
- **轨道分析**: 通过当前TLE参数（倾角、高度、偏心率）分析碎片的空间分布特征

## 💡 用例与查询设计

### Use Case 1: Collision Avoidance & Risk Monitoring
**目标**: 识别在当前或未来特定时间窗口内，距离过近的物体（卫星与碎片、或卫星与卫星），以发出碰撞预警。

#### Query 1.1: 近距离接近事件检测（Conjunction Assessment）
```sql
-- 计算并显示未来24小时内，与"国际空间站(ISS)"距离小于10公里的所有物体
-- 涉及: 时空计算、子查询、JOIN
-- 返回: 物体名称、最小距离、相对速度、预计接近时间
```
**技术要点**: 使用轨道传播算法（SGP4）计算未来位置，计算三维欧几里得距离。

#### Query 1.2: 高能碰撞风险对识别
```sql
-- 找出所有相对速度超过10km/s 且相互距离在50km以内的物体对
-- 涉及: 自连接（Self-Join）、几何计算、聚合筛选
-- 返回: 物体对ID、相对速度、当前距离、潜在碰撞能量
```
**技术要点**: 高相对速度意味着即使小碎片也能造成灾难性损伤（动能 ∝ v²）。

#### Query 1.3: 轨道密度热力图生成
```sql
-- 统计特定高度层（例如500km-550km）当前的物体密度
-- 涉及: 窗口函数、空间分桶（Spatial Binning）、密度计算
-- 返回: 高度区间、物体数量、密度（物体数/立方公里）
```
**技术要点**: 将轨道空间划分为三维网格，统计每个网格单元的物体数量。

---

### Use Case 2: Launch Window Optimization
**目标**: 帮助发射服务商寻找轨道"最干净"的时间段和路径，避免发射阶段与现有碎片发生碰撞。

#### Query 2.1: 最优发射时间窗口查询
```sql
-- 给定发射场坐标和目标轨道倾角，找出未来一周内穿过该路径的物体数量最少的时间段
-- 涉及: 时间序列分析、聚合函数、滑动窗口
-- 返回: 时间段、穿越物体数量、推荐发射窗口（绿色/黄色/红色）
```
**技术要点**: 将一周划分为1小时时间片，统计每个时间片内的轨道冲突数。

#### Query 2.2: 发射轨迹碰撞风险模拟
```sql
-- 模拟查询：如果在t0时刻发射，计算运载火箭轨迹与现有LEO物体的最小交会距离
-- 涉及: 路径积分、最小距离计算、条件筛选
-- 返回: 潜在冲突物体列表、最小距离、冲突发生高度
```
**技术要点**: 将火箭上升轨迹建模为时间函数，检测与所有在轨物体的最小距离。

#### Query 2.3: 高风险发射走廊识别
```sql
-- 识别在特定发射走廊上方，碎片云密度最高的"黑名单"高度区间
-- 涉及: 空间聚合、排序、TOP-N查询
-- 返回: 高度区间、碎片密度、主要碎片来源事件
```
**技术要点**: 某些高度层因历史解体事件（如Fengyun-1C、Cosmos-Iridium碰撞）而碎片密集。

---

### Use Case 3: Space Debris Cluster Analysis (碎片群轨道分布分析)
**目标**: 分析当前主要碎片群（如Fengyun-1C、Cosmos-2251）的轨道分布特征，评估其对不同高度层卫星的潜在威胁。

#### Query 3.1: 碎片群密度分布统计
```sql
-- 统计特定碎片群（通过名称筛选，如'FENGYUN 1C DEB'）在不同高度区间的数量分布
-- 涉及: 聚合函数、数学计算（从平均运动推算高度）、GROUP BY
-- 返回: 高度区间（如800-850km）、碎片数量、占该群总数的百分比
```
**技术要点**: 不需要历史解体数据，直接分析当前TLE数据中的轨道参数。

#### Query 3.2: 碎片群轨道面离散度分析
```sql
-- 计算碎片群的倾角（Inclination）和升交点赤经（RAAN）的标准差
-- 涉及: 统计函数（STDDEV, AVG）、子查询
-- 返回: 碎片群名称、平均倾角、倾角标准差、轨道面分布范围
```
**技术要点**: 标准差越大，说明碎片扩散得越广，威胁范围越大。

#### Query 3.3: 跨轨道层威胁评估
```sql
-- 识别碎片群的近地点（Perigee）和远地点（Apogee）覆盖的范围
-- 涉及: 极值函数（MIN, MAX）、轨道几何计算
-- 返回: 碎片群名称、最低近地点、最高远地点、威胁的高度层范围
```
**技术要点**: 使用现有TLE数据计算近地点/远地点，评估其扫过的空间体积。

---

### Use Case 4: Location-Based Visibility Prediction (基于位置的可见性预测)
**目标**: 为任意给定地理坐标（如灾区、科考站、移动终端）预测未来时间窗口内的卫星过顶情况。

#### Query 4.1: 动态位置过顶预测
```sql
-- 给定任意经纬度(lat, lon)，预测未来1小时内过顶的通信卫星
-- 涉及: 地理空间计算（动态输入坐标）、时间序列、条件筛选
-- 返回: 卫星名称、过顶开始时间、最大仰角、持续时间
```
**技术要点**: 不依赖预存的地面站表，而是将坐标作为查询参数直接输入。

#### Query 4.2: 区域覆盖密度分析
```sql
-- 计算当前时刻，覆盖特定地理区域（如北极圈）的卫星数量
-- 涉及: 空间几何判断（点在多边形内或圆形覆盖范围内）、聚合统计
-- 返回: 区域名称、覆盖卫星数、主要卫星类型
```
**技术要点**: 基于卫星星下点（Sub-satellite point）计算覆盖情况。

#### Query 4.3: 星座可视性窗口计算
```sql
-- 计算某一点（如伦敦）在未来24小时内，能看到Starlink星座中任意一颗卫星的总时长
-- 涉及: 复杂的时间区间合并（Time Interval Merging）、聚合计算
-- 返回: 地点、总可视时长、覆盖率（可视时长/24小时）
```
**技术要点**: Starlink星座有数千颗卫星，需计算其可视时间段的并集。

---

### Use Case 5: Compliance & Sustainability Reporting
**目标**: 为国际监管机构（如UN COPUOS、FCC）提供数据，评估运营商是否遵守空间可持续性准则。

#### Query 5.1: 超期在轨物体识别
```sql
-- 识别所有已超过预期寿命且状态仍标记为"在轨"但未进行离轨操作的卫星
-- 涉及: 日期计算、条件筛选、JOIN
-- 返回: 卫星名称、运营商、预期寿命、实际在轨时长、超期时间
```
**技术要点**: 根据国际空间碎片缓解指南，LEO卫星应在任务结束后25年内离轨（FCC 2022年新规则将其缩短至5年）。

#### Query 5.2: 商业公司碎片责任评估
```sql
-- 计算各商业公司的"质量/碎片比"，即其在轨总质量与产生的碎片数量的比例
-- 涉及: 多表JOIN、聚合函数（SUM）、除法计算
-- 返回: 公司名称、在轨总质量（kg）、产生碎片数量、质量/碎片比
```
**技术要点**: 该比值越高，说明公司的轨道管理越负责（质量集中在大型可控物体，而非碎片）。

#### Query 5.3: 离轨趋势与发射平衡分析
```sql
-- 分析过去10年，每年进入大气层烧毁的物体数量与新发射物体数量的趋势对比
-- 涉及: 时间聚合（按年）、窗口函数（计算年增长率）、双轴对比
-- 返回: 年份、发射数量、离轨数量、净增量、累积在轨物体数量
```
**技术要点**: 如果离轨速度远低于发射速度，轨道环境将面临"凯斯勒综合症"（Kessler Syndrome）风险。

## 🎨 项目特色与创新点

### 1. 避免主题重复
与往年项目相比，本主题首次涉及**航天领域**，完全避开了交通（Uber、公交）、娱乐（Netflix、Spotify）、金融（股票、银行）等常见领域。

### 2. 技术复杂度高
- **时空数据耦合**: 同时处理三维空间坐标（x, y, z）和时间维度（t）
- **物理计算**: 涉及轨道力学（开普勒定律、SGP4传播算法）
- **实时性要求**: 模拟卫星位置更新和碰撞检测的实时计算

### 3. 真实数据支持
所有数据源（CelesTrak、Space-Track、UCS）均为公开、权威、结构化良好的数据，且有成熟的Python库（如`skyfield`、`sgp4`、`poliastro`）支持数据处理。

### 4. 社会意义
空间碎片问题已成为全球性挑战，本项目具有实际应用价值和政策参考意义。
